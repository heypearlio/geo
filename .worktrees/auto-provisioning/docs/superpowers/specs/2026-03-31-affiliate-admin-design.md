# GEO Affiliate Admin — Design Spec

## What This Is

A scoped admin system for Misti's 1099 affiliate contractors. Each affiliate gets a personal login at `local.heypearl.io/[slug]/login` where they see only leads tagged to their landing page. Misti onboards new affiliates from her existing god admin. No email automation in this phase — focus is correct lead tagging and scoped admin access.

**This lives in the geo-landing codebase and geo Supabase project.** Not separate.

---

## The Two Systems Using This

| System | Domain | Audience |
|---|---|---|
| `/local` affiliates | `local.heypearl.io/[slug]` | 1099 contractors (e.g. Christina) |
| `/v2` real estate | `white-label-master` (separate) | Paying agent clients |

This spec covers only the `/local` affiliate system.

## Database Changes

### New table: `affiliates`

```sql
CREATE TABLE affiliates (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name             TEXT NOT NULL,
  slug             TEXT UNIQUE NOT NULL,
  tag              TEXT UNIQUE NOT NULL,
  email            TEXT,                    -- for password reset emails
  phone            TEXT,                    -- shown on their landing page
  headshot_url     TEXT,                    -- uploaded or linked headshot
  calendly_url     TEXT,                    -- their booking link
  meta_pixel_id    TEXT,                    -- Misti adds this after setup
  password_hash    TEXT,                    -- null until first login setup
  session_token    TEXT,                    -- set on login, stored in cookie
  invite_token     TEXT,                    -- nanoid, for first-login setup link
  invite_used      BOOLEAN DEFAULT false,   -- true once password is set
  reset_token      TEXT,                    -- for forgot-password flow
  reset_expires_at TIMESTAMPTZ,             -- reset token expiry (1 hour)
  active           BOOLEAN DEFAULT true,
  created_at       TIMESTAMPTZ DEFAULT now()
);
```

- `slug` = URL segment (e.g. `christina`)
- `tag` = value stored in `geo_local_submissions.source_tag` (usually same as slug)
- `invite_token` = nanoid generated by Misti during onboarding — shared as a setup link
- `password_hash` = bcrypt hash of affiliate's self-chosen password
- `email` = affiliate's email address — used only for password reset
- `headshot_url` = stored URL whether affiliate uploaded a photo (via Supabase Storage) or pasted a link directly

### New table: `geo_local_submissions`

```sql
CREATE TABLE geo_local_submissions (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email         TEXT NOT NULL,
  first_name    TEXT,
  business_type TEXT,
  source_tag    TEXT NOT NULL,
  created_at    TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX idx_geo_local_submissions_source_tag ON geo_local_submissions(source_tag);
CREATE INDEX idx_geo_local_submissions_created_at ON geo_local_submissions(created_at DESC);
```

- Every opt-in from any affiliate landing page saves here
- `source_tag` = the `funnelTag` from the landing page config (e.g. `"christina"`)
- Email sequences are NOT triggered in this phase — data only

## Auth — Password Flow

### First Login (Invite Link)

1. Misti creates affiliate in her god admin → system generates `invite_token` (nanoid)
2. God admin shows the invite link: `local.heypearl.io/[slug]/setup?token=[invite_token]`
3. Misti shares this link with the affiliate (copy button in her admin)
4. Affiliate clicks link → `/[slug]/setup` page validates token → shows password setup form
5. Affiliate sets password → system bcrypt-hashes it, stores in `password_hash`, marks `invite_used = true`, clears `invite_token`
6. Redirect to `/[slug]/leads`

### Returning Login

- Affiliate visits `local.heypearl.io/[slug]/login`
- Enters password → POST `/api/affiliate/login` with `{ slug, password }`
- Server bcrypt-compares against `password_hash`
- On match: sets httpOnly cookie `affiliate_auth` = `slug:bcrypt_session_token` (30 days)
- On fail: "Incorrect password" error

### Cookie Auth

- Cookie name: `affiliate_auth`
- Value: `slug:session_token` (session token stored in `affiliates.session_token` column — see below)
- Helper `getAffiliateFromRequest(req)` — parses cookie, looks up affiliate, returns `{ id, slug, tag, name }` or null
- Each protected API route calls this helper, returns 401 if null
- Each protected page redirects to `/[slug]/login` if not authed (client-side check)

**Session token approach** (avoids storing password in cookie): on login, generate a nanoid `session_token`, store in `affiliates.session_token`, put `slug:session_token` in cookie. Validate by looking up `affiliates WHERE slug = $1 AND session_token = $2 AND active = true`.

Add `session_token TEXT` to the affiliates table.

### Change Password

- Available inside the dashboard (settings link in nav or profile dropdown)
- `POST /api/affiliate/change-password` with `{ slug, currentPassword, newPassword }`
- Validates current password before setting new one
- Invalidates session token (forces re-login)

### Forgot Password

1. Affiliate visits `/[slug]/forgot-password`, enters their email
2. POST `/api/affiliate/forgot-password` with `{ slug, email }`
3. System looks up affiliate by slug + email match → generates `reset_token` (nanoid), sets `reset_expires_at` = now + 1 hour
4. Sends email via Resend (transactional, not a sequence): link to `/[slug]/reset-password?token=[reset_token]`
5. Affiliate clicks link → validates token + expiry → shows new password form
6. POST `/api/affiliate/reset-password` with `{ slug, token, newPassword }`
7. Sets new `password_hash`, clears `reset_token` + `reset_expires_at`, invalidates session

## Affiliate Admin Pages

### Setup (`/[slug]/setup?token=...`)

3-step wizard — validates invite token before showing any step.

**Step 1: Set your password**
- Password + confirm password fields

**Step 2: Your profile**
- Full name (pre-filled from what Misti entered, editable)
- Email address (for password recovery)
- Phone number
- Headshot — two options: upload a photo OR paste an image URL. Upload goes to Supabase Storage, returns a URL. Either way, `headshot_url` is stored.

**Step 3: Your booking link**
- Calendly URL field
- Preview of their landing page URL: `local.heypearl.io/[slug]`

On completing Step 3: all fields saved to `affiliates` table, `invite_used = true`, `invite_token` cleared, redirect to `/[slug]/leads`.

Token invalid/used/expired: "This invite link is no longer valid. Contact Misti for a new one."

### Login (`/[slug]/login`)
- Shows affiliate's name (looked up from `affiliates` table by slug)
- Password field
- "Forgot password?" link → `/[slug]/forgot-password`
- On success: sets cookie, redirects to `/[slug]/leads`
- On fail: "Incorrect password" error

### Forgot Password (`/[slug]/forgot-password`)
- Single email field
- Submit → sends reset email if email matches their record
- Always shows "If that email matches your account, you'll get a reset link" (prevents enumeration)

### Reset Password (`/[slug]/reset-password?token=...`)
- Validates token + expiry before showing form
- New password + confirm fields
- On success: redirect to login with "Password updated. Please log in."
- Token expired: "This reset link has expired. Request a new one."

### Leads (`/[slug]/leads`)
- Table: Email | Name | Business Type | Date
- Filtered to `source_tag = affiliate.tag`
- Search by email/name
- Pagination (50 per page)
- No lead detail panel in this phase

### Activity (`/[slug]/activity`)
- Total leads (all time)
- New leads this week
- New leads this month
- Recent opt-ins list (last 10)

### Calls (`/[slug]/calls`)
- Upcoming calls (from `geo_scheduled_calls` where email matches any of their leads)
- Past calls needing outcome
- Same outcome marking as Misti's admin (pending / bought / no_show / rescheduled)

### Layout Nav
- Their name + "Dashboard" branding (not "geo admin")
- Links: Leads · Activity · Calls
- Settings link (change password)
- No access to templates, funnel, campaigns, emails — those are god-only

## God Onboarding — New Affiliate

Misti adds affiliates from `geo.heypearl.io/admin/affiliates` (new page in her existing admin).

**New Affiliate form:**
- Name (e.g. "Christina")
- Slug (auto-generated from name, editable)
- Tag (defaults to slug, editable)
- Submit → inserts into `affiliates` table → generates `invite_token` → shows result panel

**Result panel (after creating):**
- Invite link: `local.heypearl.io/[slug]/setup?token=[invite_token]` + Copy button
- "Send this link to [Name] — they'll set their own password and fill in their profile."
- Reminder note (styled as a callout): "Once they've completed setup, come back here to add their Meta Pixel ID."
- Link expires only when used — no time limit

**Affiliates list:**
- Name | Slug | Status | Meta Pixel | Active toggle | Lead count
- Status: "Pending Setup" (invite not used) or "Active" (password set)
- Actions: Regenerate invite link (if pending) | Add/Edit Meta Pixel | Reset password (sends email)

**No wizard needed** — affiliates are simpler than white-label clients. One form, one row, done.

---

## Lead Ownership

Misti owns all leads. Affiliates are 1099 contractors with view access only.

- Affiliates can **see** their tagged leads and call outcomes
- Affiliates **cannot** export, delete, suppress, or view other affiliates' leads
- Call outcome marking (attended/no-show) is allowed — they ran the call
- All data stays in geo Supabase under Misti's account
